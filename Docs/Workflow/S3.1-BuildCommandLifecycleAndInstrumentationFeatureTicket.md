# S3.1 Build Command Lifecycle and Instrumentation Feature Ticket

- **Title:** S3.1 Build Command Lifecycle and Instrumentation Feature Ticket
- **Scope:** This document records the Definition of Ready feature ticket for roadmap step `S3.1`, covering Build command entry-point behavior, orchestration sequencing, run metadata capture, structured stage logging, and Build-summary instrumentation expectations for the AxionFrame baseline.

## Purpose

This document applies the Stage 1.2 feature-ticket format to roadmap step `S3.1` from `roadmap.md`.

It translates the approved Stage 1 and Stage 2 documentation baseline into an execution-ready work item for the first executable Build workflow and its shared instrumentation foundation.

## Feature DoR Record

- **Feature ID:** `AXF-S3.1-BUILD-LIFECYCLE`
- **Feature Title:** Build command lifecycle and structured instrumentation baseline
- **Roadmap Step:** `S3.1`
- **Owner:** SW
- **Primary Domain:** `Frame`
- **Supporting Core Service Areas:** `Validation`, `Infrastructure`
- **Implementation Scope Summary:** Implement the Build command entry point and shared runtime orchestration so the Add-in can load parameters, execute pre-Build validation, invoke downstream generation stages, capture run metadata (`run-id`, timestamp, config hash), and emit deterministic stage logs and a Build completion summary. This ticket is anchored to `Frame` because frame generation is the first downstream Build consumer in the Stage 3 delivery sequence, while the orchestration and instrumentation capability itself must remain shared across all documented domains.

#### Configuration Readiness

- **Configuration Keys:** `frame.layout.primary.*`, `frame.profile.selection.*`, `frame.naming.ruleSet`, `pivot.geometry.primary.*`, `pivot.hole.strategy.*`, `pivot.naming.mates`, `height.supportedConfigurations.values`, `height.indexing.activation.*`, `height.validation.*`, `plateBrace.dimensions.primary.*`, `plateBrace.export.dxfEligible`, `plateBrace.naming.ruleSet`, `validation.*`, `run.packageOutputs`
- **Schema/Placeholder Status:** `Defined`
- **Default Value Notes:** Build lifecycle execution should consume the documented defaults and normalized configuration behavior already established in Stage 2.2. Run metadata should record `run-id`, timestamp, and config hash without changing the approved configuration baseline.
- **Range / Limit / Tolerance Notes:** This ticket does not introduce new geometry limits; it must preserve the documented frame, pivot, height, and plate/brace ranges and the existing validation enforcement rules. Build-stage logging must occur in the documented sequence `load -> validate -> generate parts -> generate assembly -> summary`.

#### Output Expectations

- **Expected CAD Outputs:** Build-triggered part and assembly generation stages for frame, pivot, height-adjust, and plate/brace workflows, with supported height configuration activation included in Build scope.
- **Expected Final Outputs:** Structured run logs and a Build completion summary; no STEP, DXF, BOM, or packaged release artifacts are required directly by this step.
- **Expected Report Sections / Trace Points:** `load`, `validate`, `generate parts`, `generate assembly`, `summary`, plus run metadata fields `run-id`, `timestamp`, and `config hash`

#### Planned Tests

- **Unit Tests Planned:** Run-metadata creation; config-hash stability for identical normalized inputs; stage-order enforcement for Build logging; summary payload field presence.
- **Integration Tests Planned:** Execute the Build command through the Add-in host path and confirm parameter loading, pre-Build validation, ordered stage logging, and Build-summary emission.
- **Regression Checks Planned:** Perform three consecutive Build runs with the same baseline configuration and confirm no unhandled exceptions and stable stage-order output across runs.
- **Manual Verification Planned:** Review the Build completion summary in SolidWorks and inspect the generated log path and excerpt against the lifecycle sequence defined in `Docs/Workflow/ProjectLifecycle.md`.

#### Risks and Dependencies

- **Blocking Dependencies:** `Gate 2` is complete and provides the configuration, validator, and naming baseline required before Build workflow delivery can proceed. Downstream geometry modules `S3.2`, `S3.3`, and `S3.4` are not yet complete, so this step must establish orchestration boundaries and logging hooks that those modules can attach to later.
- **Known Risks:** Build instrumentation may drift from the ME review checkpoints if stage names and summary content are not centralized early; host-layer command wiring may become tightly coupled to runtime logging unless orchestration and diagnostics remain in the core boundary defined by the architecture and ownership docs.
- **Open Questions:** Whether the first Build completion summary should remain in the current Add-in UI surface or move immediately into a dedicated shared runtime/reporting abstraction while preserving the existing host/core split.

#### Ready Decision

- **DoR Status:** `Ready`
- **SW Reviewer:** Recorded in repository docs
- **ME Reviewer:** Recorded in repository docs
- **Decision Basis:** The lifecycle sequence, architecture ownership, configuration/validation baseline, and roadmap acceptance criteria define enough scope to begin Build-lifecycle implementation without unresolved product-scope gaps.
