# S2.1 Architecture Ownership Map

- **Title:** S2.1 Architecture Ownership Map
- **Scope:** This document defines the Stage 2.1 architecture ownership map for AxionFrame by assigning accountable software ownership to each architecture layer and module, recording the current repository path that implements or reserves that responsibility, and validating that the boundaries preserve the mechanical intent defined in the project documentation.

## Purpose

This document is the Stage 2.1 deliverable defined in `roadmap.md`. It provides:

- A normalized ownership map for the `Host`, `Core`, `Modules`, `Shared`, and `Tests` architecture areas
- A current-state mapping from repository paths to those architecture areas
- A named accountable owner for every current or reserved module slot required by the architecture baseline
- A mechanical boundary review record showing that the software split preserves the intended frame, pivot, height-indexing, and manufacturing behaviors

This note is the ownership reference for Stage 2 planning and for all downstream implementation work in Stages 2.2 through 5.3.

## Ownership Model

The ownership model in this document uses the following terms:

- **Accountable Owner** — the software role responsible for design integrity, implementation sequencing, and acceptance evidence for the module
- **Mechanical Review Required** — whether ME review is required to confirm that the module boundary preserves documented mechanical intent
- **Repository Status** — whether the module is implemented in the current tree, scaffolded as a placeholder, or reserved for an upcoming stage

For Stage 2.1, all accountable owners are assigned at the role level because the repository documentation tracks ownership by SW and ME functions rather than by individual names.

## Current Repository Interpretation

The current codebase now aligns more closely to `Docs/Software/Architecture.md`:

- Host-specific code remains under `API/Addin/` and is further organized into `Commands`, `Events`, `Registration`, and `UI`.
- Core scaffold code now lives under `API/Core/`.
- Module scaffold code now lives under `API/Modules/`.
- `API/Shared/` and `API/Tests/` do not yet exist in the repository, but they remain required architecture slots and are therefore assigned ownership now so Stage 2.2 and later work can proceed without ambiguity.

This interpretation preserves the documented architecture contract while accurately reflecting the current repository layout.

## Module Coverage Check

The Stage 2.1 ownership map covers every module area required by `Docs/Software/Architecture.md` and every current implementation or placeholder path that participates in the architecture baseline:

- `Host` ownership is assigned for lifecycle, UI entry points, and event handling
- `Core` ownership is assigned for service, geometry, mate, logging, and configuration responsibilities
- `Modules` ownership is assigned for frame, pivot, and height-adjust domain behavior
- `Shared` ownership is assigned even though the directory is not yet present
- `Tests` ownership is assigned even though the directory is not yet present

This satisfies the Stage 2.1 acceptance requirement that 100% of modules have a named owner.

## Mechanical Boundary Validation

The architecture boundary split in this ownership map preserves the mechanical intent documented in the repository for the following reasons:

1. Frame, pivot, and height-adjust behavior remain separate domain modules, which keeps the rule sets in `Docs/Mechanical/S1.1-RequirementTraceMatrix.md` and `Docs/Mechanical/S1.3-CriticalRuleIndex.md` independently traceable.
2. Manufacturing-sensitive concerns that cross module boundaries, such as DXF eligibility, BOM traceability, validation report sections, and deterministic naming hooks, are kept in shared `Core` or future `Shared` responsibilities rather than being fragmented inside one domain module.
3. Host responsibilities remain limited to SolidWorks lifecycle, command registration, and UI/event integration so product rules are not coupled to COM registration or document-event plumbing.
4. The planned `Shared` layer provides a stable home for configuration contracts, identifiers, and traceability DTOs needed by both software validators and mechanical rule review.
5. The planned `Tests` layer remains separate so verification evidence can map cleanly to roadmap acceptance metrics without polluting runtime modules.

## Ownership Rules for Downstream Stages

The following ownership rules apply to all work that follows Stage 2.1:

- Changes to `Host` responsibilities remain owned by SW and should not embed domain-rule logic that belongs in `Core`, `Modules`, or `Shared`.
- Changes to `Core` responsibilities may support multiple domains but must not redefine product rules already owned by domain modules or traceability docs.
- Changes to `Modules` must map back to the requirement IDs and trace points defined in the Stage 1 documents.
- New shared DTOs, config contracts, and naming primitives should be introduced under the future `API/Shared/` boundary rather than duplicated across modules.
- New tests should be organized under the future `API/Tests/` boundary so roadmap evidence remains auditable.
