# S2.2 Configuration Schema and Validators Feature Ticket

- **Title:** S2.2 Configuration Schema and Validators Feature Ticket
- **Scope:** This document records the Definition of Ready feature ticket for roadmap step `S2.2`, covering configuration contracts, default resolution, schema validators, cross-field validators, and deterministic validation message behavior for the AxionFrame baseline.

## Purpose

This document applies the Stage 1.2 feature-ticket format to roadmap step `S2.2` from `roadmap.md`.

It translates the approved Stage 1 and Stage 2.1 documentation baseline into an implementation-ready work item for the first substantive coding phase of the project.

## Feature DoR Record

- **Feature ID:** `AXF-S2.2-CFG-VALIDATORS`
- **Feature Title:** Configuration schema and deterministic validators baseline
- **Roadmap Step:** `S2.2`
- **Owner:** SW (lead), ME (review)
- **Primary Domain:** `Height Indexing`
- **Supporting Core Service Areas:** `Validation`, `Infrastructure`
- **Implementation Scope Summary:** Implement the baseline configuration contract, default-resolution behavior, deterministic schema validation, cross-field validation, and validator result contracts for all approved frame, pivot, height-indexing, plate/brace, export, validation, and run-metadata settings. This ticket is a shared runtime capability, anchored to `Height Indexing` because the release-blocking safety rule `HGT-003` is the highest-risk validation constraint in the current baseline.

#### Configuration Readiness

- **Configuration Keys:** `frame.layout.primary.*`, `frame.profile.selection.*`, `frame.naming.ruleSet`, `pivot.geometry.primary.*`, `pivot.hole.strategy.*`, `pivot.naming.mates`, `height.supportedConfigurations.values`, `height.indexing.activation.*`, `height.validation.*`, `plateBrace.dimensions.primary.*`, `plateBrace.export.dxfEligible`, `plateBrace.naming.ruleSet`, `exports.*`, `validation.*`, `run.packageOutputs`
- **Schema/Placeholder Status:** `Defined`
- **Default Value Notes:** Defaults are fully specified in `Docs/Software/ConfigurationSchemaSpecification.md` and should be applied before cross-field validation. The initial implementation should treat `Config/GlobalParams.json` as the bootstrap source for persisted baseline values while preserving deterministic fallback to documented defaults.
- **Range / Limit / Tolerance Notes:** Frame extent `620.0-980.0 mm`; pivot-axis location `300.0-450.0 mm`; pivot-hole diameter `10.5-11.0 mm`; supported heights `680.0 mm`, `730.0 mm`, `780.0 mm`; height tolerance `+/-1.0 mm`; plate thickness `5.0-8.0 mm`; strict cross-field rules must enforce ordered min/max bounds, height-set equality, required height count, DXF eligibility dependency, and `StrictRelease` stop-on-critical-failure behavior.

#### Output Expectations

- **Expected CAD Outputs:** `N/A` for direct geometry generation; this ticket enables validated Build inputs for downstream CAD work.
- **Expected Final Outputs:** Deterministic normalized configuration payloads and deterministic validation result sets that downstream Build and Final Output flows can consume.
- **Expected Report Sections / Trace Points:** `validation.schema`, `validation.crossField`, `traceability.ruleCoverage`, validator IDs `VAL-FRM-001-LAYOUT`, `VAL-FRM-002-PROFILE`, `VAL-FRM-003-NAMING`, `VAL-PVT-001-GEOMETRY`, `VAL-PVT-002-HOLES`, `VAL-PVT-003-MATES`, `VAL-HGT-001-SUPPORTED-CONFIGS`, `VAL-HGT-002-ACTIVATION`, `VAL-HGT-003-HEIGHT-VALIDITY`, `VAL-PLT-001-DIMENSIONS`, `VAL-PLT-002-DXF-ELIGIBILITY`, `VAL-PLT-003-NAMING`

#### Planned Tests

- **Unit Tests Planned:** Default resolution; required-field checks; numeric-range checks; enum validation; array validation; cross-field rule enforcement; deterministic validator ordering; deterministic message formatting; strict-rule blocking behavior.
- **Integration Tests Planned:** Load `Config/GlobalParams.json` into a normalized configuration object; execute pre-Build validation from the future core orchestration path without requiring live geometry generation.
- **Regression Checks Planned:** Re-run identical valid and invalid configurations and confirm stable `ruleId`, `validatorId`, `message`, `expected`, and `actual` outputs.
- **Manual Verification Planned:** Cross-check encoded defaults, ranges, and strict-rule mappings against `Docs/Mechanical/S1.1-RequirementTraceMatrix.md`, `Docs/Mechanical/S1.3-CriticalRuleIndex.md`, and the configuration and validation specifications.

#### Manufacturability Readiness Checks

- **Manufacturability Review Required:** `Yes`
- **Manufacturing-Relevant Geometry Identified:** `Yes`
- **Fabrication/Export Impact Identified:** `Yes`
- **Tolerance or Fit Review Status:** `Ready`
- **ME Reviewer / Input Source:** Mechanical review against `Docs/Mechanical/DesignAndManufacturing.md`, `Docs/Mechanical/S1.1-RequirementTraceMatrix.md`, and `Docs/Mechanical/S1.3-CriticalRuleIndex.md`

#### Risks and Dependencies

- **Blocking Dependencies:** `S2.1` is complete; implementation should introduce the planned shared-contract and test boundaries before validator logic expands across later stages.
- **Known Risks:** `Config/GlobalParams.json` is currently empty; repository structure does not yet include the documented `API/Shared/` and `API/Tests/` layers; naming-warning validator scope can drift into `S2.3` if strict boundary rules are not maintained.
- **Open Questions:** Whether the first persisted config shape should be nested JSON or a flat key-path representation; whether naming warnings for `FRM-003`, `PVT-003`, and `PLT-003` are fully implemented in `S2.2` or stubbed until `S2.3` applies naming logic to generated CAD entities.

#### Ready Decision

- **DoR Status:** `Ready with Constraints`
- **SW Reviewer:** Recorded in repository docs
- **ME Reviewer:** Recorded in repository docs
- **Decision Basis:** The architecture ownership map, schema specification, validation specification, trace matrix, and critical-rule index define enough scope to start implementation. The ticket remains constrained only by unresolved implementation-shape details for persisted config layout and by the need to keep naming-application work separated from `S2.3`.
