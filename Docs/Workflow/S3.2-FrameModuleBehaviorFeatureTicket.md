# S3.2 Frame Module Behavior Feature Ticket

- **Title:** S3.2 Frame Module Behavior Feature Ticket
- **Scope:** This document records the Definition of Ready feature ticket for roadmap step `S3.2`, covering baseline frame layout generation, approved frame-profile application, deterministic frame feature naming, Build-stage integration expectations, and frame-specific verification targets for the AxionFrame baseline.

## Purpose

This document applies the Stage 1.2 feature-ticket format to roadmap step `S3.2` from `roadmap.md`.

It translates the approved Stage 1, Stage 2, and Stage 3.1 documentation baseline into an execution-ready work item for the first domain-level CAD behavior delivery in the Build workflow.

## Feature DoR Record

- **Feature ID:** `AXF-S3.2-FRM-BASELINE`
- **Feature Title:** Baseline frame module geometry and deterministic behavior
- **Roadmap Step:** `S3.2`
- **Owner:** SW (lead), ME (review)
- **Primary Domain:** `Frame`
- **Supporting Core Service Areas:** `Validation`, `Infrastructure`
- **Implementation Scope Summary:** Implement the documented frame rules inside the domain module layer so the Build workflow can generate baseline frame geometry, apply approved frame profiles, and expose deterministic frame naming and traceability outputs. This ticket must attach to the completed S3.1 Build orchestration baseline without expanding into pivot, height-adjustment, plate/brace, or Final Output scope.

#### Requirement References

- **Requirement IDs:** `FRM-001`, `FRM-002`, `FRM-003`
- **Source Documents:** `Docs/Mechanical/DesignAndManufacturing.md`, `Docs/Mechanical/S1.1-RequirementTraceMatrix.md`, `Docs/Mechanical/S1.3-CriticalRuleIndex.md`, `Docs/Software/Architecture.md`, `Docs/Software/ConfigurationSchemaSpecification.md`, `Docs/Software/DeterministicNamingStandard.md`, `Docs/Software/ValidationAndErrorHandlingSpecification.md`, `Docs/Workflow/ProjectLifecycle.md`, `Docs/Workflow/S3.1-BuildCommandLifecycleAndInstrumentationFeatureTicket.md`
- **Related Trace Matrix Rows:** `FRM-001`, `FRM-002`, `FRM-003`

#### Configuration Readiness

- **Configuration Keys:** `frame.layout.primary.memberExtentMin`, `frame.layout.primary.memberExtentMax`, `frame.layout.primary.placementTolerance`, `frame.profile.selection.allowedProfiles`, `frame.profile.selection.dimensionTolerance`, `frame.naming.ruleSet`
- **Schema/Placeholder Status:** `Defined`
- **Default Value Notes:** The approved frame defaults already exist in the Stage 2.2 schema baseline and should be consumed through the normalized configuration path established by the current core workflow. The initial implementation should treat the documented allowed profile set as the canonical baseline and must not introduce alternate frame-profile families outside the approved configuration contract.
- **Range / Limit / Tolerance Notes:** Primary frame member extents must remain within `620.0-980.0 mm`; placement tolerance must remain within `+/-0.5 mm`; allowed baseline profiles are `40x40x2.0_SHS` and `60x30x2.0_RHS`; profile dimensional tolerance remains `+/-0.2 mm`; `frame.layout.primary.memberExtentMin` must remain less than or equal to `frame.layout.primary.memberExtentMax` as enforced by the existing validator contract.

#### Output Expectations

- **Expected CAD Outputs:** Generated frame layout feature aligned to `AXF_FRM_LAYOUT_PRIMARY`; generated frame profile feature aligned to `AXF_FRM_PROFILE_MAIN`; frame-driven part and assembly outputs for baseline Build runs.
- **Expected Final Outputs:** BOM trace input for `frame.members` and validation report sections for `frame.layout` and `traceability.naming.frame`; this step does not directly require STEP, DXF, packaged outputs, or release artifacts.
- **Expected Report Sections / Trace Points:** `frame.layout`, `frame.members`, `traceability.naming.frame`, `AXF_FRM_LAYOUT_PRIMARY`, `AXF_FRM_PROFILE_MAIN`

#### Planned Tests

- **Unit Tests Planned:** Frame-module baseline output naming; frame config-to-output mapping for extents and profile selection; deterministic repeated-output checks for frame feature identifiers.
- **Integration Tests Planned:** Execute the Build workflow with the baseline configuration and confirm the frame module contributes the expected part artifacts, naming hooks, and stage-summary evidence through the S3.1 orchestration path.
- **Regression Checks Planned:** Re-run identical baseline Build executions and confirm frame artifact identity, frame feature naming, and frame-related summary traces remain unchanged across repeated runs.
- **Manual Verification Planned:** Review generated frame geometry in SolidWorks against the baseline frame extent, placement, and profile rules and confirm the dimension checklist expected by ME review.

#### Manufacturability Readiness Checks

- **Manufacturability Review Required:** `Yes`
- **Manufacturing-Relevant Geometry Identified:** `Yes`
- **Fabrication/Export Impact Identified:** `Yes`
- **Tolerance or Fit Review Status:** `Ready`
- **ME Reviewer / Input Source:** Mechanical review against `Docs/Mechanical/DesignAndManufacturing.md`, `Docs/Mechanical/S1.1-RequirementTraceMatrix.md`, `Docs/Mechanical/S1.3-CriticalRuleIndex.md`, and the frame-related Build expectations in `Docs/Workflow/ProjectLifecycle.md`

#### Risks and Dependencies

- **Blocking Dependencies:** `S3.1` is complete and provides the Build command wiring, run metadata, and structured stage instrumentation required for frame delivery. `S2.2` and `S2.3` are complete and provide the approved configuration, validation, and deterministic naming baseline that frame behavior must consume. Downstream steps `S3.3`, `S3.4`, and `S3.5` depend on stable frame references but are outside this ticket's direct scope.
- **Known Risks:** The frame module may remain a naming-only placeholder if geometry behavior is not attached to the documented frame rules; frame profile selection may drift from BOM and validation trace points if profile logic is duplicated outside the domain module; frame-specific logic may become over-coupled to the core orchestration path unless ownership stays in the `API/Modules` boundary defined by the architecture.
- **Open Questions:** Whether the first S3.2 implementation should emit only the baseline frame primitives required for downstream pivot integration or include the full initial frame-member output set in the first pass.

#### Ready Decision

- **DoR Status:** `Ready`
- **SW Reviewer:** Recorded in repository docs
- **ME Reviewer:** Recorded in repository docs
- **Decision Basis:** The frame requirement rows, configuration keys, validator mappings, deterministic naming hooks, Build-stage integration baseline, and manufacturability constraints are defined well enough to begin implementation without unresolved product-scope gaps.
