# S1.2 Feature Definition of Ready Template

- **Title:** S1.2 Feature Definition of Ready Template
- **Scope:** This document defines the Stage 1.2 Definition of Ready template for AxionFrame feature work, including mandatory requirement references, configuration keys, expected outputs, planned tests, and manufacturability readiness checks.

## Purpose

This document is the Stage 1.2 deliverable defined in `roadmap.md`. It provides:

- The standard Definition of Ready template for implementation tickets
- The mandatory readiness fields required before work may move to `In Progress`
- Manufacturability readiness checks for cross-functional review
- Two completed sample feature tickets that demonstrate template usage

This template is aligned to the target AxionFrame documentation baseline and approved platform/toolchain. It is intended to be reused for roadmap implementation work without changing field names or removing required checks.

## Definition of Ready Policy

A feature is ready for implementation only when all mandatory fields in this template are completed and the ticket has been reviewed for both software readiness and manufacturability readiness where applicable.

No implementation ticket may enter `In Progress` until this template is completed.

## Required Field Set

Every feature ticket must include the following mandatory fields:

- Requirement references
- Configuration keys
- Expected outputs
- Planned tests
- Manufacturability readiness checks

The canonical domain model for this template is:

- **Product domain modules:** `Frame`, `Pivot`, `Height Indexing`, `Plate and Brace`
- **Core service areas:** `Exports`, `Validation`, `Infrastructure`

When a ticket spans both categories, the product domain remains the primary domain and any supporting core-service concern should be recorded separately.

## Field Guidance

### Requirement References Guidance

- Requirement IDs should reference roadmap-backed documentation, especially `Docs/Mechanical/S1.1-RequirementTraceMatrix.md`
- Source documents should point to the specific docs that define the expected behavior
- Related trace matrix rows should use stable IDs such as `FRM-001` or `PVT-002`

### Domain Classification Guidance

- Use **Primary Domain** for the product subsystem that owns the mechanical or CAD behavior being delivered
- Use **Supporting Core Service Areas** for cross-domain concerns such as export flow, validation flow, or shared infrastructure
- Do not use `Exports`, `Validation`, or `Infrastructure` as substitutes for the product domain when the ticket is ultimately delivering frame, pivot, height-indexing, or plate/brace behavior
- If a ticket exists only to support shared runtime capability, set **Primary Domain** to the product area most directly affected and explain the shared-service scope in the implementation summary

### Configuration Readiness Guidance

- Configuration keys may be placeholders when Stage 2 schema work has not yet been implemented
- Missing keys must be explicitly called out rather than omitted
- Range and tolerance notes should use either defined values or controlled placeholders from the trace matrix

### Output Expectations Guidance

- Expected CAD outputs should define what must exist after `Build`
- Expected final outputs should define what must exist after `Final Output`
- Trace points should reference the expected report section, BOM section, export class, or naming hook

### Planned Tests Guidance

- At least one verification activity must be listed
- Testing should follow the repository’s `Unit`, `Integration`, and `Regression` conventions where relevant
- Manual verification is acceptable for early stages when automated tests do not yet exist

### Manufacturability Guidance

- Any feature affecting frame members, pivot joints, height indexing behavior, or plate/brace fabrication should set manufacturability review to `Yes`
- If manufacturability input is missing, the feature should not be considered fully ready unless the `Ready with Constraints` status is justified

## Completed Sample Ticket A

### Feature DoR Record

- **Feature ID:** `AXF-S3.2-FRM-BASELINE`
- **Feature Title:** Baseline frame geometry generation
- **Roadmap Step:** `S3.2`
- **Owner:** SW (lead), ME (review)
- **Primary Domain:** `Frame`
- **Supporting Core Service Areas:** `Validation`
- **Implementation Scope Summary:** Implement the baseline frame member layout and profile selection behavior for generated table geometry.

#### Requirement References

- **Requirement IDs:** `FRM-001`, `FRM-002`, `FRM-003`
- **Source Documents:** `Docs/Mechanical/DesignAndManufacturing.md`, `Docs/Mechanical/S1.1-RequirementTraceMatrix.md`, `Docs/Workflow/ProjectLifecycle.md`
- **Related Trace Matrix Rows:** `FRM-001`, `FRM-002`, `FRM-003`

#### Configuration Readiness

- **Configuration Keys:** `frame.layout.primary`, `frame.profile.selection`, `frame.naming.ruleSet`
- **Schema/Placeholder Status:** `Defined`
- **Default Value Notes:** Default values should implement the approved baseline frame layout and profile set from `Docs/Mechanical/DesignAndManufacturing.md`
- **Range / Limit / Tolerance Notes:** Primary structural member extents `620-980 mm`; placement tolerance `+/-0.5 mm`; allowed profile set `40 x 40 x 2.0 mm SHS`, `60 x 30 x 2.0 mm RHS`; profile dimensional tolerance `+/-0.2 mm`

#### Output Expectations

- **Expected CAD Outputs:** Generated frame parts and frame assembly features
- **Expected Final Outputs:** BOM trace for frame members; validation report frame section
- **Expected Report Sections / Trace Points:** `frame.layout`, `frame.members`, `traceability.naming.frame`

#### Planned Tests

- **Unit Tests Planned:** Naming rule verification for frame feature identifiers
- **Integration Tests Planned:** Build flow execution for baseline frame generation
- **Regression Checks Planned:** Repeat-run naming stability check
- **Manual Verification Planned:** SolidWorks review of generated frame geometry and member placement

#### Manufacturability Readiness Checks

- **Manufacturability Review Required:** `Yes`
- **Manufacturing-Relevant Geometry Identified:** `Yes`
- **Fabrication/Export Impact Identified:** `Yes`
- **Tolerance or Fit Review Status:** `Ready`
- **ME Reviewer / Input Source:** Mechanical team review against frame member and profile rules

#### Risks and Dependencies

- **Blocking Dependencies:** None within the approved baseline specification
- **Known Risks:** Downstream changes to the approved baseline profile set would require validator and BOM updates
- **Open Questions:** Whether future product variants require additional frame profile families beyond the baseline release

#### Ready Decision

- **DoR Status:** `Ready`
- **SW Reviewer:** Recorded in Docs
- **ME Reviewer:** Recorded in Docs
- **Decision Basis:** Requirement references, approved baseline values, outputs, tests, and manufacturability checks are defined for implementation without unresolved mechanical placeholders

- **Owner:** SW
- **Primary Domain:** `Plate and Brace`
- **Supporting Core Service Areas:** `Exports`
- **Implementation Scope Summary:** Implement DXF export eligibility and output handling for fabrication-relevant plate components.

#### Requirement References

- **Requirement IDs:** `PLT-001`, `PLT-002`, `PLT-003`

- **Related Trace Matrix Rows:** `PLT-001`, `PLT-002`, `PLT-003`

#### Configuration Readiness

- **Configuration Keys:** `plateBrace.dimensions.primary`, `plateBrace.export.dxfEligible`, `plateBrace.naming.ruleSet`
- **Schema/Placeholder Status:** `Defined`
- **Default Value Notes:** Export behavior should default to the approved baseline plate and brace manufacturing rules from `Docs/Mechanical/DesignAndManufacturing.md`
- **Range / Limit / Tolerance Notes:** Plate thickness range `5.0-8.0 mm`; dimensional tolerance `+/-0.2 mm`

#### Output Expectations

- **Expected CAD Outputs:** Plate and brace model entities identified as DXF export candidates
- **Expected Final Outputs:** DXF files for plate components in final output flow
- **Expected Report Sections / Trace Points:** `plateBrace.flatPatterns`, `dxf.exports`, `traceability.naming.plateBrace`

#### Planned Tests

- **Unit Tests Planned:** Export-eligibility selection logic for plate components
- **Integration Tests Planned:** Final Output DXF generation workflow
- **Regression Checks Planned:** Stability check across repeated export runs
- **Manual Verification Planned:** Review generated DXF set for required fabrication candidates

#### Manufacturability Readiness Checks

- **Manufacturability Review Required:** `Yes`
- **Manufacturing-Relevant Geometry Identified:** `Yes`
- **Fabrication/Export Impact Identified:** `Yes`
- **Tolerance or Fit Review Status:** `Needs ME Input`
- **ME Reviewer / Input Source:** Mechanical review of plate fabrication relevance and dimensional readiness

#### Risks and Dependencies

- **Blocking Dependencies:** Final DXF selection criteria depend on finalized plate definitions
- **Known Risks:** Missing manufacturable plate classification could create incomplete fabrication output
- **Open Questions:** Whether non-plate brace elements are included in the first DXF export scope

#### Ready Decision

- **DoR Status:** `Ready with Constraints`
- **SW Reviewer:** Recorded in Docs
- **ME Reviewer:** Recorded in Docs
- **Decision Basis:** The ticket includes requirement references, configuration placeholders, output expectations, planned tests, and manufacturability checks sufficient to begin implementation planning
