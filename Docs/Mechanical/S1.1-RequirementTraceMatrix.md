# Requirement Traceability Matrix

- **Title:** Requirement Traceability Matrix
- **Scope:** This document defines the Stage 1.1 requirement traceability matrix for AxionFrame by mapping mechanical product rules to configuration-key placeholders, deterministic feature or mate naming hooks, measurable control fields, and expected output/report trace points.

## Purpose

This document is the Stage 1.1 deliverable defined in `roadmap.md`. It provides:

- The initial requirement matrix for frame, pivot, height indexing, and plate/brace geometry
- Configuration-key placeholders for each documented rule
- Deterministic feature or mate naming hooks for software traceability
- Measurable range, limit, and tolerance placeholders for later schema and validation work

This matrix is derived from the current repository documentation and declared project toolchain. It establishes traceability structure without claiming that all downstream implementation or quantitative engineering values already exist in source code.

## Traceability Conventions

### Requirement ID Format

- `FRM-*` for frame rules
- `PVT-*` for pivot rules
- `HGT-*` for height-indexing rules
- `PLT-*` for plate and brace rules

### Criticality Tags

- **Critical** — required for function, manufacturability, or release acceptance
- **Supporting** — required for completeness but not currently classified as release-critical

### Configuration Placeholder Rules

- Configuration keys are placeholders for Stage 2 schema work.
- Placeholder keys are intentionally stable and machine-friendly.
- Dot-separated keys reflect expected ownership in future configuration contracts.

### Deterministic Naming Hook Rules

- Part features use `AXF_<domain>_<component>_<descriptor>`
- Assembly mates use `AXF_MATE_<domain>_<descriptor>`
- Height states use `AXF_CFG_<descriptor>`

These hooks are traceability targets for later implementation and naming standardization work.

### Quantitative Placeholder Rules

- Units are expressed explicitly as `mm`, `deg`, or `count`
- Where the current documentation does not define final numbers, the measurable field uses a controlled placeholder token rather than an empty cell
- Placeholder tokens beginning with `TBD-ME-` require mechanical value definition in later stages
